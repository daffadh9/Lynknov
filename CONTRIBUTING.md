# Contributing

## Contribution Model

Lynknov is currently a **founder-led development project**.

At this stage, the repository is publicly visible for:

- portfolio presentation
- progress visibility
- structured project communication
- documentation reference
- future collaboration readiness

This does **not** currently mean unrestricted open contribution.

External collaboration may be considered selectively in the future, but the project is not yet operating as a broadly open-source community repository.

---

## Before You Contribute

Please review the following first:

- [`README.md`](./README.md)
- [`PROJECT_STATUS.md`](./PROJECT_STATUS.md)
- [`USAGE_POLICY.md`](./USAGE_POLICY.md)
- [`SECURITY.md`](./SECURITY.md)
- [`docs/documentation/README.md`](./docs/documentation/README.md)

These documents explain the current project state, usage boundaries, and documentation structure.

---

## What Is Welcome Right Now

At the current stage, the most appropriate forms of contribution or collaboration interest are:

- thoughtful feedback
- issue reporting
- architecture discussion
- product critique
- documentation suggestions
- UI/UX observations
- selective implementation discussion with permission

If the repository later becomes more collaboration-ready, this file can evolve accordingly.

---

## What Is Not Assumed

Please do not assume that public visibility means:

- unrestricted reuse is allowed
- direct commercial reuse is allowed
- large code contributions will automatically be merged
- derivative implementations are authorized

See [`USAGE_POLICY.md`](./USAGE_POLICY.md) for the current repository usage boundaries.

---

## Contribution Principles

Any future contribution to Lynknov should aim to be:

- aligned with the product direction
- structurally clean
- scoped appropriately
- respectful of the existing repo organization
- well-documented when necessary
- understandable by future collaborators

Contributions should improve clarity, not create new ambiguity.

---

## Preferred Areas of Help

The most helpful contribution categories generally include:

### Documentation quality
- improving clarity
- fixing ambiguity
- strengthening structure
- aligning related docs

### Product thinking and review
- identifying inconsistencies
- questioning scope decisions
- improving prioritization logic
- improving user flow reasoning

### UI and UX review
- reducing friction
- improving information hierarchy
- strengthening conversion logic
- improving consistency

### Technical quality
- identifying maintainability concerns
- pointing out architecture risks
- improving structure without unnecessary complexity

---

## Before Opening an Issue

Before opening an issue, please check:

- whether the topic is already documented
- whether the concern is actually a security issue
- whether the suggestion fits the current project phase
- whether the issue is clear, scoped, and actionable

Use the provided GitHub issue templates where available.

---

## Before Opening a Pull Request

At the current phase, pull requests should generally be considered only when:

- the change is clearly valuable
- the scope is limited and understandable
- the change aligns with the product direction
- the change does not create structural drift
- documentation is updated when needed

Large unsolicited PRs are usually less helpful than well-scoped and well-explained changes.

---

## Contribution Style Expectations

If contributing in any form, please aim for:

- clear naming
- concise explanations
- high signal-to-noise
- strong structure
- product awareness
- respect for the existing documentation system

Good contribution is not only about correctness.  
It is also about fit, clarity, and maintainability.

---

## Documentation Expectations

If a contribution changes:

- product direction
- developer workflow
- architecture
- onboarding expectations
- repository structure

then related documentation should be updated as well.

Lynknov treats documentation as part of product quality, not as optional cleanup.

---

## Security Issues

Please do **not** report security vulnerabilities through normal public issues.

See [`SECURITY.md`](./SECURITY.md) for the private reporting process.

---

## Collaboration Requests

If you are interested in deeper collaboration, selective contribution, or authorized project involvement, please contact the repository owner directly first.

This is especially important for:

- substantial code contributions
- design-system work
- architecture changes
- derivative usage requests
- educational or portfolio-related reuse requests

---

## Review Standard

Any accepted contribution should ideally meet this standard:

- it improves the project meaningfully
- it respects product direction
- it does not create unnecessary complexity
- it is understandable by someone new to the repo
- it strengthens the project rather than fragmenting it

---

## Final Note

Lynknov is being built intentionally.

That means contributions are appreciated most when they support:

- clarity
- structure
- maintainability
- product alignment
- documentation quality
- long-term coherence

Thank you for respecting the project, its stage of development, and the systems being built around it.
