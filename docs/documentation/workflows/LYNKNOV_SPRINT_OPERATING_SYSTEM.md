# Lynknov Sprint Operating System

## Purpose

This document defines a lightweight sprint operating model for Lynknov.

It is designed for founder-led or small-team execution where priorities need to stay focused and visible without excessive process overhead.

---

## Sprint Philosophy

A sprint in Lynknov should exist to create momentum with clarity.

That means each sprint should answer:
- what matters now,
- what is being built,
- what “done” means,
- and what should not be touched yet.

---

## Suggested Sprint Duration

Recommended duration:
- **1 week** for fast-moving solo execution
- **2 weeks** when coordination or larger scope requires more room

---

## Sprint Template

Each sprint should define:

### 1. Sprint goal
Example:
- finalize landing page structure
- ship auth page foundation
- build dashboard shell

### 2. Core deliverables
List the concrete outputs expected by sprint end.

### 3. Non-goals
Clarify what is intentionally out of scope.

### 4. Quality checks
Define what quality means for this sprint:
- responsive
- visually aligned
- documented
- technically clean enough

### 5. Review outcome
At the end of the sprint, capture:
- what shipped
- what slipped
- what was learned
- what should happen next

---

## Sprint Discipline Rules

- Do not overload the sprint with unrelated ideas
- Protect the main goal
- Prefer finishing fewer things well over starting too many things
- Update documentation if sprint decisions change product understanding

---

## Recommended Rhythm

### Start of sprint
- confirm priority
- confirm deliverables
- review relevant docs

### During sprint
- build in focused passes
- commit regularly
- keep decisions traceable

### End of sprint
- review output
- log progress
- define the next sprint clearly

---

## Final Principle

A good sprint should leave Lynknov more complete, more coherent, and easier to continue.
